📓 4.3.2.1 Hosting with Firebase

In addition to providing database and authentication functionality, Firebase also provides free hosting (at least for smaller sites). As a site scales up, Firebase services do have a cost. However, this is a great solution for smaller portfolio sites. If you do end up building a site that scales, that can be a good problem to have - perhaps you have the beginnings of a start-up. If so, a cloud-based solution like Firebase will help with many of the headaches that scaling up can cause, ranging from efficient database queries to fast hosting.

In this lesson, we'll cover the steps necessary to host a site using Firebase. Fortunately, it's very easy to deploy with Firebase CLI. This will be a global install:

npm install -g firebase-tools@7.13.1

To actually use the Firebase CLI, we need to log in with the following command in the terminal:

firebase login

Provide the user name and password you use to sign into the Firebase console.

We can verify we are connected by running the following command:

firebase projects:list

This will list out all of our Firebase projects.

Next, we need to run firebase init in the directory of the project we want to hook up to Firebase. This will initialize a new Firebase project. We'll see the following screen in the terminal:

We have to use the space bar to select options. We'll select "Hosting: Configure and deploy Firebase Hosting." Then push Enter to confirm.

Next, you can select either a new or existing project. We'll assume that we actually have a project we want to deploy - so select "Use an existing project."

Next, we can select which existing project to deploy.

For the question What do you want to use as your public directory?, type in build. (You can also just hit enter and change the configuration file later, which we'll discuss soon.)

Next, for the question Configure as a single-page app (rewrite all urls to /index.html)?, select No.

Finally, for the question File public/index.html already exists. Overwrite?, select No.

At this point, Firebase will automatically generate firebase.json and .firebaserc files for us.

Your firebase.json file should look similar to this one (but may not match exactly):

firebase.json
{
  "functions": {
    "predeploy": [
      "npm --prefix \"$RESOURCE_DIR\" run lint"
    ],
    "source": "functions"
  },
  "hosting": {
    "public": "build",
    "ignore": [
      "firebase.json",
      "**/.*",
      "**/node_modules/**"
    ]
  }
}

Note that "public" needs to have a value of "build" because create-react-app will put all of our assets in a build directory when we run npm run build. If it has a different value in your firebase.json, you can change it now.

Next, we can run npm run build (assuming our project is ready to go). This will create a bundle in the build directory of our project (create-react-app's default behavior).

We are almost ready to deploy. However, it's a good idea to make sure everything looks good before we deploy. We can do this by running firebase serve. This will run our project at http://localhost:5000. You can make sure that everything looks just as you expect.

Once you are done, you can shut down the local server and then type in firebase deploy.

Next, go to [PROJECT-NAME].firebaseapp.com/, where [PROJECT-NAME] is the name of your project. Note that the name should be exactly as it's stated in the Firebase console. For instance, if you were deploying the help queue, the URL for the deployed site might look something like this: https://redux-help-queue-e53d7.firebaseapp.com/. (Note that this is just an example link and does not actually go to a deployed site.)

You may also want to add a custom domain, especially if this is a portfolio site. For more information on setting up your own domain, check out Firebase's custom domain documentation.

Finally, if your deployed site is connected to a database, it's highly recommended that you take a closer look at Firebase's security rules. Remember how the default rules that Firebase provides allow full read/write access for just thirty days? This is intentional. Having full read/write access while developing a site that no one else can access is fine. However, once the site is deployed, it's a good idea to take a closer look at the security settings.
